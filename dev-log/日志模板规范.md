# dev-log 日志模板规范

本文件由 day28 的最终版日志作为样板沉淀而成，用于约束后续 day29-36 重写时的统一风格与详尽程度。任何新写或重写的 dev-log，都应满足下方"必须项 (MUST)"，"应当项 (SHOULD)"为最佳实践。

---

## 1. 文档骨架

每篇 dev-log 顶层结构固定：

```
# Day NN — 标题（用 / 分隔的关键改进）

## 0. 阅读指南
## 1. 引言（这一天为什么要做这些改动）
## 2. 改进 A — <名字>
## 3. 改进 B — <名字>
...
## 9. 工程化收尾（gitignore / CI / 构建脚本变化等）
## 10. 验证（构建命令 + ctest 输出 + 端到端 curl 验证）
## 11. 局限与下一步（明确说明遗留问题与 day NN+1 计划）
```

每个 "改进 X" 子节进一步固定为：

```
### X.1 业务场景        — 必须用具体可观测的事故 / 指标说话
### X.2 接口/数据结构    — 摘抄头文件原文
### X.3 纯策略函数       — 摘抄实现原文，强调"无副作用"
### X.4 嵌入执行路径     — 摘抄调用方原文，说明"在什么时机调"
### X.5 全流程追踪       — 见本规范第 3 节
### X.6 单元测试如何覆盖  — 摘抄 test/*.cpp 关键断言
```

---

## 2. 代码引用规则（MUST）

1. **任何代码块必须原封不动摘抄自源文件**，不得伪代码、不得改写、不得简化。
2. **每个代码块上方必须以"来自 [显示路径](相对路径)"形式注明出处**。例：
   `来自 [HISTORY/day28/common/Connection.cpp](HISTORY/day28/common/Connection.cpp)：`
3. 若需要省略中间行，用 `...` 单行表示，且不可跨函数。
4. 若同一函数被两处引用，可分两个独立代码块；不要拼接成"伪整段"。

---

## 3. 全流程追踪格式（核心）

这是 day28 的标志性升级，用来让读者真正"理解"每条新 API 的运行逻辑，而非仅仅"知道"它存在。

### 3.1 必须包含的四块内容

每个 `X.5 全流程追踪` 子节由下面四块按序组成：

#### 第 1 块：业务场景 + 时序总览表

- 一段 1-3 行的"业务场景"描述：用具体数字（缓冲大小 / 连接数 / RSS / 客户端类型）说清"这条流程在生产中长什么样"。
- 一张时序总览表，列至少 5 行（T0..T4 或 轮 1..轮 N），每行包含至少 4 列：
  - 时刻 / 轮次
  - 业务动作（external trigger）
  - 关键状态量的取值（例如 `outputBuffer = 16.06 MB`、`epoll 内部 fd=42 = 在`、`buf->readableBytes() = 38`）
  - 决策结果 / 副作用

#### 第 2 块：逐函数代码追踪（核心新规范）

按时序总览表中"决策点"为节奏，每个决策点开一个 `#### 第 N 步：xxx` 小标题，内部固定结构：

1. 一句话说明"现在进入了哪个函数 / 哪一段"。
2. **代码块**：摘抄该函数对应的源代码（见第 2 节规则）。
3. **代入实参**：用文字 + 公式形式列出此刻每个参数 / 成员变量的真实数值。例：
   > 传入实参：`bufferedBytes = 16.06 MB`、`readPaused = false`、`cfg = {4MB, 16MB, 64MB}`。
4. **判断分支走向**：以列表形式逐条复盘 if 条件的 true/false 与最终选择的分支。例：
   > - `16.06 MB > 64 MB`？否 → 跳过 close 分支。
   > - `!false && 16.06 MB > 16 MB`？是 → `d.shouldPauseRead = true`。
5. **副作用清单 + 此刻全局状态**：用 fenced code block 列出本步执行后系统进入的快照状态：
   ```
   outputBuffer_              = 16.06 MB
   readPausedByBackpressure_  = true
   epoll 监听位               = EPOLLOUT
   ```
6. **(可选)** 一段"为什么这样设计"的注解，把决策与设计意图绑定。

#### 第 3 块：状态机全景 / ASCII 图（如适用）

若改进涉及多状态切换（如回压、连接生命周期、半包→完整包的状态机），必须画一个 ASCII 状态图，标出阈值、转移条件、副作用动作。

#### 第 4 块：各函数职责一句话表

最后用一张表收尾，列出本次涉及的全部函数 / 角色，每个一句话说明"在哪个时机调用 + 它的核心职责"。例：

| 函数 | 调用时机 | 职责 |
|------|---------|------|
| `evaluateBackpressure` | applyBackpressure / tryResume 内部 | 纯判断，无副作用 |
| `applyBackpressureAfterAppend` | send() 之后 | 暂停读 / 触发兜底断连 |

### 3.2 反面例子（禁止出现）

- ❌ "解出 /a 完整请求 → retrieve(18) → 回调 → reset" 这种**单行高度抽象**的流程描述。
- ❌ 时序表只列动作，不列状态量数值。
- ❌ "完美。" 这种没有信息量的总结。
- ❌ 用伪代码 / 注释片段代替源代码。

---

## 4. 业务场景写作要求

- **MUST 用真实可观测事件作为入口**，例如：
  - "凌晨 3 点 OOM 告警"
  - "ab -n 100 -c 1 -k 中 50 条请求超时"
  - "服务器跑 7 天后 epoll_ctl(ADD) 返回 EEXIST"
  - "树莓派 4G 弱网下载 2 GB 文件"
- 不要用泛泛的 "为了支持高并发" / "为了更好的可维护性" 之类的空话。
- 必要时给一个具体配置数字（例如 high=16MB、maxConnections=10000），让后文追踪能直接代入。

---

## 5. 末尾"局限与下一步"格式

- 列出本日改动的真实遗留（如"回压只覆盖 OutputBuffer 单方向"）。
- 末尾一行用自然语言衔接到下一天："接下来 Day NN+1 会..."。

- 文件采用 LF 换行、UTF-8 无 BOM。
